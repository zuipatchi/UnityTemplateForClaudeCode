# PRD作成スキル

このスキルは、Unity個人ゲーム開発向けの高品質な企画・要求定義書(Game PRD)を作成するためのガイドです。

## 前提条件

PRD作成を開始する前に、以下が完了している必要があります:

### アイデアの壁打ちが完了している

作りたいゲームのアイデアについて、Claude Codeとの対話を通じてブラッシュアップを完了している必要があります。

### docs/ideas/initial-requirements.md が作成されている

**重要**: 壁打ちの内容は、ユーザーが以下のファイルに保存する必要があります:

**ファイルパス**: `docs/ideas/initial-requirements.md`

このファイルには以下の内容が含まれている必要があります:
- ゲームの基本的なアイデアとコンセプト
- ゲーム体験・面白さの核
- ターゲットプレイヤーの概要
- 実装したい主要ゲームプレイ機能
- MVPの範囲（最初にリリースする形）

PRD作成時には、このファイルの内容を参照して詳細化を行います。

## 既存ドキュメントの優先順位

**重要**: `docs/product-requirements.md` に既存のPRDがある場合、
以下の優先順位に従ってください:

1. **既存のPRD (`docs/product-requirements.md`)** - 最優先
   - プロジェクト固有の要件が記載されている
   - このスキルのガイドより優先する

2. **このスキルのガイド** - 参考資料
   - 汎用的なテンプレートと例
   - 既存PRDがない場合、または補足として使用

**新規作成時**: このスキルのテンプレートとガイドを参照
**更新時**: 既存PRDの構造と内容を維持しながら更新

## 出力先

作成したPRDは以下に保存してください:

```
docs/product-requirements.md
```

PRDを作成する際は、次のテンプレートを使用してください: ./template.md

## PRD作成プロセス

### 1. initial-requirements.md の確認

まず、ユーザーが作成した初期要求仕様を確認します:

```bash
Read('docs/ideas/initial-requirements.md')
```

### 2. PRDドラフトの生成

initial-requirements.mdの内容をもとに、テンプレートに従ってPRDを生成します。

### 3. PRDのレビューと改善

生成されたPRDを以下の観点でレビューします:

#### レビュー観点

1. ゲームコンセプト(コアループ・面白さの核)は明確か
2. ターゲットプレイヤーは具体的か
3. 成功指標は測定可能か
4. ゲームプレイ機能は実装可能なレベルまで詳細化されているか
5. 技術要件(Unity固有の制約など)は網羅されているか

#### レビュー結果の評価基準

生成したPRDは以下の形式で評価します:

**✅ 強み**
- コアループが明確で面白さの核が具体的に記述されている
- 機能仕様が実装レベルまで詳細化されている
- スコープがMVPとして現実的な範囲に収まっている

**⚠️ 改善が必要な点**

機能要件の曖昧さ:
- 問題: 具体的なゲームプレイ仕様が不明確な箇所がある
- 推奨: 具体的な操作仕様やゲームルールを明記

成功指標の測定方法:
- 問題: 測定方法が不明確
- 推奨: プレイテストでの検証方法を明記

### 4. レビュー後の改善

レビューで指摘された問題を一つずつ確認し、具体化が必要な箇所を改善します:

1. 指摘された問題を一つずつ確認
2. 具体化が必要な箇所を改善
3. 改善後、再度レビューを実施
4. 問題がなくなるまで繰り返す

## PRD作成の重要なポイント

### 1. 具体性と測定可能性

すべての要件は具体的で測定可能でなければなりません。

**悪い例**:
- ゲームは楽しい必要がある
- 操作は直感的であるべき

**良い例**:
- 1セッションのプレイ時間: 15〜30分を想定
- 初めてのプレイヤーがチュートリアルなしで基本操作を5分以内に把握できる

### 2. コアループの明確化

すべての機能はコアループから逆算して定義します。

**コアループのフォーマット**:
```
[アクション] → [フィードバック] → [報酬・進行] → [次のアクションへ]
```

**例**:
```
敵を倒す → エフェクト・スコア加算 → レベルアップ・新スキル解放 → より強い敵へ挑む
```

### 3. 優先順位の明確化

- **P0(必須)**: MVPに含める機能。これがないとゲームとして成立しない
- **P1(重要)**: 初期リリース後すぐに追加すべき機能
- **P2(できれば)**: 将来的に追加を検討する機能

個人開発では **P0のスコープを極力絞る** ことが重要です。

## PRDの主要セクション詳細

### 1. プロダクト概要

#### 構成要素

1. **タイトル**: ゲーム名とキャッチコピー
2. **ジャンル**: ゲームジャンル（例: 2Dアクション、パズル、ローグライク）
3. **プラットフォーム**: 対象プラットフォーム（例: PC(Windows/Mac)、WebGL）
4. **ゲームコンセプト**: 3つの主要コンセプト
5. **ゲームビジョン**: 目指す体験を3〜5文で
6. **エレベーターピッチ**: 「〇〇のような△△だが、□□が特徴のゲーム」形式で1文

#### 例
```markdown
### タイトル
**DungeonSprint** - 10秒で判断する即興ダンジョン探索

### ジャンル
ローグライク / ターン制アクション

### プラットフォーム
PC(Windows) / WebGL

### ゲームコンセプト
- 超短期ループ: 1ランが5〜10分で完結する爽快なテンポ感
- 即興の意思決定: 毎ターン制限時間内に行動を選ぶ緊張感
- ビルド構築の楽しさ: スキルの組み合わせで毎回異なる戦略

### ゲームビジョン
「今日も少しだけ遊ぼう」と思ったら30分経っていた——そんなゲームを目指す。
複雑な操作を覚えなくてもすぐ遊べるが、深く研究するほど強くなれる。

### エレベーターピッチ
Slay the Spireのようなデッキ構築ローグライクだが、各ターンに制限時間を設けることでリアルタイムの緊張感を加えたゲーム。
```

### 2. ターゲットプレイヤー

#### 必須要素

1. **基本属性**: 年齢層、ゲーム経験
2. **好きなゲームの傾向**
3. **プレイシーン**: どこで・どれくらいの時間プレイするか
4. **求める体験**: どんな感情・達成感を求めているか

#### 例
```markdown
### ターゲットプレイヤー
- 20〜35歳、ゲーム歴5年以上のカジュアル〜ミッドコアゲーマー
- ローグライクやインディーゲームが好き
- 隙間時間(15〜30分)にサクッと遊びたい
- 「強くなっていく感覚」と「試行錯誤の楽しさ」を求めている
```
### 3. 完成の定義
- P0機能がすべて動作する
- 自分が1周クリアまで遊べる
- 初見の友人1人がルール説明なしで遊べる

### 4. ゲームプレイ機能要件

#### コア機能(MVP)

各機能には以下を含めます:
- プレイヤー体験の説明
- 受け入れ条件（チェックリスト形式）
- 優先度(P0/P1/P2)

**フォーマット**:
```markdown
### [機能名]

プレイヤー体験:
[プレイヤー]が[状況]で[行動]すると[体験・感情]が生まれる

受け入れ条件:
- [ ] 条件1
- [ ] 条件2

優先度: P0 / P1 / P2
```

#### ゲームプレイ仕様の記述例
```markdown
### 移動システム

プレイヤー体験:
プレイヤーがキャラクターを動かすとき、レスポンシブな操作感でストレスなく動かせる

受け入れ条件:
- [ ] WASDまたは矢印キーで8方向移動できる
- [ ] 入力から移動開始まで1フレーム以内に反応する
- [ ] 壁との衝突判定が自然に感じられる

優先度: P0

---

### スコアシステム

プレイヤー体験:
敵を倒すたびにスコアが加算され、達成感とランキングへの競争意識が生まれる

受け入れ条件:
- [ ] 敵撃破ごとにスコアが加算される
- [ ] ゲームオーバー時にスコアが表示される
- [ ] ハイスコアがローカル保存される(PlayerPrefs使用)

優先度: P0
```

### 5. 技術要件(Unity固有)
```markdown
### 開発環境
- Unityバージョン: 〇〇.〇.〇f1(LTS推奨)
- レンダーパイプライン: URP / Built-in(どちらか明記)
- ターゲット解像度: 1920x1080(16:9基準)
- フレームレート目標: 60fps

### アーキテクチャ方針
- シーン構成: (例) Boot / MainMenu / Gameplay / Result
- データ管理: ScriptableObject でゲームパラメータを管理
- セーブデータ: PlayerPrefs(小規模) / JSONファイル(中規模)

### アセット方針
- 自作 / 無料アセット / 有料アセットは使用しない
- フォント、BGM、SEの調達方針

### パフォーマンス要件
- ターゲット実行環境の最低スペック(例: Intel HD Graphics 以上)
- Draw Call 目標値(例: 100以下)
```

### 6. 非ゲームプレイ要件
```markdown
### UI/UX
- メインメニュー: スタート・オプション・終了
- ポーズ機能: Escキーで一時停止
- ゲームオーバー画面: スコア表示・リトライ・メニューへ戻る

### オーディオ
- BGM: ループ再生、音量調整可能
- SE: 操作・攻撃・ダメージ・ゲームオーバーの最低4種

### ローカライズ
- 対応言語: 日本語のみ(MVP) / 英語追加(P2)
```

## 品質基準とチェックポイント

PRDの品質を確保するため、以下のチェックポイントを確認してください:

### ゲームコンセプト
- [ ] コアループが「アクション→フィードバック→報酬→次のアクション」で記述されているか
- [ ] エレベーターピッチが1文で言えるか
- [ ] 面白さの核(コアバリュー)が明確か

### ターゲットプレイヤー
- [ ] 具体的なプレイシーンが定義されているか
- [ ] 求める体験・感情が記述されているか

### スコープ管理
- [ ] P0機能だけでゲームとして最低限成立するか
- [ ] MVP完成までの期間が現実的か(個人開発で1〜3ヶ月が目安)
- [ ] P2機能が「やりたいこと」の肥大化になっていないか

### 技術要件
- [ ] Unityバージョンが明記されているか
- [ ] シーン構成が定義されているか
- [ ] アセット調達方針が明確か

### ゲームプレイ機能
- [ ] すべての機能がプレイヤー体験ベースで記述されているか
- [ ] 受け入れ条件が実装・テスト可能な粒度か
- [ ] 優先度(P0/P1/P2)が設定されているか

## まとめ

PRD作成の成功のポイント:

1. **コアループを最初に定義する**: 面白さの核が曖昧なまま機能を増やさない
2. **スコープを絞る**: 個人開発のMVPはP0だけで十分。P2は「ドリームリスト」として割り切る
3. **プレイヤー体験ベースで書く**: 実装仕様ではなく「プレイヤーがどう感じるか」から逆算する
4. **Unity固有の制約を明記する**: バージョン・パイプライン・アセット方針は早めに決める
